### Vorschlag 2: REG_SZ/REG_EXPAND_SZ UTF-16LE Bug fixen

**Kategorie:** Anwendung & Code
**Prioritaet:** Hoch
**Aufwand:** Klein

**Ist-Zustand:** `dumper.rs:90` verwendet `String::from_utf8()` fuer Registry-Strings. Windows speichert REG_SZ als UTF-16LE. Der Code produziert fuer Nicht-ASCII-Werte falsche Ergebnisse.

**Soll-Zustand:** Korrekte UTF-16LE-Dekodierung mit `String::from_utf16()` oder den Decode-Hilfsfunktionen der winreg-Crate.

**Umsetzung:**
1. In `match_value_to_json()` den REG_SZ-Arm aendern:
   ```rust
   REG_SZ | REG_EXPAND_SZ => {
       let wide: Vec<u16> = val.bytes.chunks_exact(2)
           .map(|c| u16::from_le_bytes([c[0], c[1]]))
           .collect();
       let s = String::from_utf16(&wide)
           .map(|s| s.trim_end_matches('\0').to_string())
           .map(RegValueJson::String)
           .unwrap_or(RegValueJson::Invalid { reason: "invalid UTF-16".into() });
       s
   }
   ```
2. Unit-Test fuer UTF-16LE-Strings hinzufuegen
3. Hinweis: Bei Update auf winreg 0.56 koennte dieses Problem ggf. durch die neue API geloest werden
